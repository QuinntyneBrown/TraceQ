# TraceQ — L2 Requirements

---

## L1-1: CSV Import Pipeline

### L2-1.1: CSV Parsing Engine

The system shall parse CSV files exported from Windchill PLM, recognizing and mapping the following standard columns (case-insensitive matching, ignoring leading/trailing whitespace):

| Windchill Column | Domain Field | Required |
|---|---|---|
| Number | `RequirementNumber` | Yes |
| Name | `Name` | Yes |
| Description | `Description` | No |
| Type | `Type` | No |
| State | `State` | No |
| Priority | `Priority` | No |
| Owner | `Owner` | No |
| Created On | `CreatedDate` | No |
| Modified On | `ModifiedDate` | No |
| Module | `Module` | No |
| Parent Number | `ParentNumber` | No |
| Traced To | `TracedTo` | No |

The parser shall handle:
- Quoted fields containing commas and newlines (RFC 4180)
- UTF-8 and UTF-8 BOM encoded files
- Files with unmapped columns (ignored gracefully)
- Files missing optional columns

**Acceptance Criteria:**
1. A CSV with all 12 columns parses into a list of requirement DTOs with all fields populated.
2. A CSV with only Number and Name columns parses successfully with optional fields as null/empty.
3. A CSV with quoted fields containing commas and embedded newlines parses correctly.
4. A CSV with unmapped extra columns (e.g., "Custom Field 1") parses without error, ignoring the extra columns.
5. Column matching is case-insensitive (e.g., "NUMBER", "number", "Number" all map to `RequirementNumber`).

### L2-1.2: Import Service with Upsert Semantics

The system shall provide an import service that persists parsed requirements to the SQLite metadata store using upsert semantics: if a requirement with the same `RequirementNumber` already exists, it shall be updated; otherwise, it shall be inserted. The service shall track import batches with a timestamp and file name, and return a summary (inserted count, updated count, error count, skipped count).

**Acceptance Criteria:**
1. Importing a CSV with 100 new requirements inserts all 100 and returns inserted=100, updated=0.
2. Re-importing the same CSV returns inserted=0, updated=100 (or updated=0 if no fields changed, with a skipped count).
3. Importing a CSV where 50 requirements exist and 50 are new returns inserted=50, updated=50.
4. Each import creates an `ImportBatch` record with timestamp, filename, and summary counts.
5. A requirement with an invalid or missing `RequirementNumber` is counted as an error and does not prevent other requirements from importing.

### L2-1.3: Import API Endpoint

The system shall expose a `POST /api/import/csv` endpoint that accepts a multipart file upload of a CSV file, invokes the import pipeline, and returns the import summary as JSON. The endpoint shall validate the uploaded file (must be .csv, max 50 MB) and return appropriate HTTP status codes (200 for success, 400 for validation errors, 422 for parse errors).

**Acceptance Criteria:**
1. Uploading a valid CSV via `POST /api/import/csv` returns 200 with a JSON body containing inserted/updated/error counts.
2. Uploading a non-CSV file returns 400 with an error message.
3. Uploading a file exceeding 50 MB returns 400 with a size-limit error message.
4. Uploading an empty file returns 422 with a meaningful error.

### L2-1.4: Import History & Status API

The system shall expose `GET /api/import/history` returning a paginated list of past import batches (timestamp, filename, counts) and `GET /api/import/{batchId}` returning details for a specific batch including per-requirement status (inserted/updated/error with error messages).

**Acceptance Criteria:**
1. `GET /api/import/history` returns a paginated JSON array of import batches sorted by timestamp descending.
2. `GET /api/import/{batchId}` returns the batch summary plus a list of per-requirement outcomes.
3. Pagination supports `page` and `pageSize` query parameters with defaults (page=1, pageSize=20).

## L1-2: Local Embedding Generation

### L2-2.1: ONNX Model Integration

The system shall load the all-MiniLM-L6-v2 ONNX model at application startup using `Microsoft.ML.OnnxRuntime` and expose an `IEmbeddingService` interface with a method `Task<float[]> GenerateEmbeddingAsync(string text)` that tokenizes input text and runs inference to produce a 384-dimensional float vector. The model file shall be bundled with the application (not downloaded at runtime).

**Acceptance Criteria:**
1. `GenerateEmbeddingAsync("test input")` returns a `float[]` of length 384.
2. The same input always produces the same embedding (deterministic).
3. The embedding vector is L2-normalized (magnitude ≈ 1.0).
4. The model loads from a local file path configured in `appsettings.json` with no network calls.
5. If the model file is missing at startup, the application logs a clear error and throws a startup exception.

### L2-2.2: Text Tokenization

The system shall implement a WordPiece tokenizer compatible with the all-MiniLM-L6-v2 vocabulary, handling:
- Lowercasing
- Unicode normalization
- Subword tokenization with `##` prefixes
- Special tokens (`[CLS]`, `[SEP]`, `[PAD]`)
- Truncation to max 256 tokens
- Attention mask generation

The tokenizer vocabulary shall be bundled as a local file.

**Acceptance Criteria:**
1. Tokenizing "Hello world" produces correct token IDs matching the reference Python tokenizer output.
2. Input longer than 256 tokens is truncated to 256 tokens (including `[CLS]` and `[SEP]`).
3. The tokenizer handles Unicode characters (e.g., accented characters, symbols) without crashing.
4. Attention masks correctly mark real tokens as 1 and padding as 0.

### L2-2.3: Batch Embedding Pipeline

The system shall provide a batch embedding method `Task<Dictionary<string, float[]>> GenerateBatchEmbeddingsAsync(IEnumerable<(string id, string text)> items)` that processes multiple texts efficiently. After each CSV import, the system shall automatically generate embeddings for all newly inserted or updated requirements by concatenating `Name + " " + Description` as the embedding input text.

**Acceptance Criteria:**
1. After importing a CSV with 100 requirements, all 100 have embeddings generated and stored.
2. Re-importing a CSV where 10 requirements changed regenerates embeddings only for those 10.
3. Batch embedding of 500 requirements completes without out-of-memory errors.
4. A requirement with an empty description uses only the Name for embedding generation.

## L1-3: Vector Storage & Indexing

### L2-3.1: Qdrant Client Configuration

The system shall configure a Qdrant .NET client (`Qdrant.Client`) to connect to a locally-running Qdrant instance (default: `localhost:6334`). Connection parameters (host, port, gRPC port) shall be configurable via `appsettings.json`. The system shall create a collection named `requirements` at startup if it does not exist, configured for 384-dimensional vectors with cosine distance.

**Acceptance Criteria:**
1. On first startup with a running Qdrant instance, the `requirements` collection is created with vector size 384 and cosine distance metric.
2. On subsequent startups, the existing collection is reused without error.
3. If Qdrant is unreachable, the application logs a warning and operates in degraded mode (metadata-only, no vector search).
4. Connection parameters are read from `appsettings.json` under `Qdrant:Host`, `Qdrant:HttpPort`, `Qdrant:GrpcPort`.

### L2-3.2: Vector Upsert & Synchronization

The system shall upsert requirement vectors into Qdrant using the requirement's database ID as the point ID. Each point shall carry a payload with `requirementNumber`, `name`, `type`, `state`, `priority`, `module`, and `owner` for server-side filtering. After embedding generation, vectors are upserted to Qdrant in batches of up to 100 points.

**Acceptance Criteria:**
1. After importing and embedding 100 requirements, Qdrant contains 100 points with correct vectors and payloads.
2. Re-importing and re-embedding an updated requirement updates the vector and payload in Qdrant without creating a duplicate point.
3. Payload fields are searchable via Qdrant's filtering API.
4. Batch upsert of 500 points completes without timeout errors.

### L2-3.3: Vector Deletion

The system shall delete vectors from Qdrant when requirements are deleted from the metadata store. A `DELETE /api/requirements/{id}` endpoint shall remove both the SQLite record and the Qdrant point.

**Acceptance Criteria:**
1. Deleting a requirement via API removes its SQLite row and its Qdrant point.
2. Deleting a non-existent requirement returns 404.
3. If Qdrant is unavailable during deletion, the SQLite row is still deleted and the orphaned vector is cleaned up on next sync.

## L1-4: Semantic Search & Query

### L2-4.1: Semantic Search API

The system shall expose `POST /api/search` accepting a JSON body `{ "query": "string", "top": int, "filters": { "type": "string?", "state": "string?", "priority": "string?", "module": "string?", "owner": "string?" } }`. The endpoint shall embed the query text using the local ONNX model, search Qdrant for the top-N nearest vectors (default N=20, max 100), apply any provided metadata filters, and return results with similarity scores and full requirement metadata from SQLite.

**Acceptance Criteria:**
1. Searching for "thermal protection" returns requirements related to thermal systems ranked by similarity score.
2. Applying a filter `{ "type": "Functional" }` returns only functional requirements.
3. Results include `similarityScore` (0.0–1.0), `requirementNumber`, `name`, `description`, and all metadata fields.
4. Requesting `top: 5` returns at most 5 results.
5. An empty query string returns 400.

### L2-4.2: Similar Requirements API

The system shall expose `GET /api/requirements/{id}/similar?top=N` that retrieves the vector for the given requirement and searches for the N most similar other requirements (excluding itself). This enables duplicate detection and traceability analysis.

**Acceptance Criteria:**
1. Calling `/api/requirements/{id}/similar?top=5` returns 5 requirements most similar to the given one.
2. The source requirement is excluded from results.
3. If the requirement has no vector (embedding not yet generated), returns 404 with a message.
4. Results include similarity scores.

### L2-4.3: Faceted Metadata API

The system shall expose `GET /api/requirements/facets` returning the distinct values and counts for each filterable field (Type, State, Priority, Module, Owner) from the SQLite metadata store. This powers filter dropdowns in the UI.

**Acceptance Criteria:**
1. Response includes arrays of `{ "value": "string", "count": int }` for each facet field.
2. Facets reflect the current state of the database (not cached stale data).
3. Null/empty values are grouped under a "(Blank)" label.

### L2-4.4: Full-Text Keyword Search

The system shall support a keyword search fallback via `GET /api/requirements?q=keyword&page=1&pageSize=20` that performs a LIKE search across `RequirementNumber`, `Name`, and `Description` in SQLite. This complements semantic search for exact-match scenarios.

**Acceptance Criteria:**
1. Searching `?q=REQ-001` returns the requirement with number REQ-001.
2. Searching `?q=thermal` returns all requirements containing "thermal" in name or description.
3. Results are paginated with total count in the response.
4. An empty `q` parameter returns all requirements (paginated).

## L1-5: Reporting & Visualization Dashboard

### L2-5.1: Dashboard Layout Engine

The system shall provide a dashboard page using `angular-gridster2` where users can add, remove, resize, and drag widgets on a responsive grid. Each widget has a type (chart, table, search, etc.), a title, and configuration. The grid shall have a minimum of 12 columns and support widget collision detection.

**Acceptance Criteria:**
1. Dashboard renders a grid with at least one default widget on first load.
2. Widgets can be dragged to a new position and the layout persists across page reloads (stored in localStorage).
3. Widgets can be resized by dragging the resize handle.
4. A "Reset Layout" button restores the default widget arrangement.
5. An "Add Widget" button opens a dialog to choose widget type and add it to the grid.

### L2-5.2: Requirement Distribution Charts

The system shall provide Chart.js-based widgets for:
- **By Type**: Doughnut chart showing requirement count per Type.
- **By State**: Horizontal bar chart showing requirement count per State.
- **By Priority**: Pie chart showing requirement count per Priority.
- **By Module**: Bar chart showing requirement count per Module (top 15).

Each chart shall fetch data from `GET /api/reports/distribution/{field}`.

**Acceptance Criteria:**
1. Each chart renders with real data from the API.
2. Charts update when new requirements are imported (manual refresh button on each widget).
3. Clicking a chart segment filters the requirements list to that category.
4. Charts handle empty datasets gracefully (display "No data" message).

### L2-5.3: Traceability Coverage Report

The system shall provide a report widget showing traceability coverage: percentage of requirements that have at least one `TracedTo` link, a list of requirements with no traceability links, and a bar chart of trace-link density (how many links each requirement has). Data from `GET /api/reports/traceability`.

**Acceptance Criteria:**
1. The report shows a coverage percentage (e.g., "78% of requirements have trace links").
2. A table lists requirements with zero trace links, sortable by Module and Type.
3. A bar chart shows the distribution of trace-link counts (0, 1, 2, 3+).
4. The report refreshes when data changes.

### L2-5.4: Similarity Cluster Report

The system shall provide a report widget that identifies clusters of highly similar requirements (similarity > 0.85) as potential duplicates or related groups. Data from `GET /api/reports/similarity-clusters?threshold=0.85`. Results are displayed as a table of clusters, each showing the requirements in the cluster and their pairwise similarity scores.

**Acceptance Criteria:**
1. The report groups requirements with similarity > threshold into clusters.
2. Each cluster shows 2+ requirements with their pairwise similarity scores.
3. The threshold is configurable via a slider (0.7–0.99, default 0.85).
4. Requirements not in any cluster are omitted from the report.

### L2-5.5: Dashboard Layout Persistence API

The system shall expose `GET /api/dashboard/layouts` and `POST /api/dashboard/layouts` to save and retrieve named dashboard layouts (grid positions, widget types, widget configs) in SQLite. `DELETE /api/dashboard/layouts/{id}` removes a saved layout.

**Acceptance Criteria:**
1. Saving a layout with a name stores the full grid state and widget configurations.
2. Loading a layout by name restores all widgets to their saved positions and configurations.
3. Multiple named layouts can be saved and listed.
4. Deleting a layout removes it from the database.

## L1-6: Web Application Shell

### L2-6.1: Angular Project Scaffold

The system shall provide an Angular 17+ application with:
- Angular Material component library installed and configured with a dark/light theme toggle
- Routing with lazy-loaded feature modules: Dashboard, Search, Import, Requirements
- A shared module for common components (header, sidebar, loading spinner)
- `ng2-charts` (Chart.js wrapper) installed
- `angular-gridster2` installed
- `HttpClient` configured with a base URL from environment config pointing to the local API

**Acceptance Criteria:**
1. `ng serve` starts the application without errors.
2. Routes `/dashboard`, `/search`, `/import`, `/requirements` each load their respective modules.
3. Angular Material theme is applied with a toggle between dark and light modes.
4. Environment config contains `apiBaseUrl` defaulting to `http://localhost:5000/api`.

### L2-6.2: Navigation & Layout

The system shall provide a responsive layout with:
- A Material toolbar header displaying the application name "TraceQ" and navigation links
- A Material sidenav on the left with navigation items: Dashboard, Search, Import, Requirements
- A main content area that renders the active route
- The sidenav collapses to icons on narrow viewports (< 960px)

**Acceptance Criteria:**
1. The header displays "TraceQ" and is visible on all routes.
2. Clicking a sidenav item navigates to the correct route and highlights the active item.
3. On viewports under 960px, the sidenav collapses to icon-only mode.
4. The layout does not produce horizontal scrollbars at any standard viewport width.

### L2-6.3: Import Page

The system shall provide an Import page with:
- A drag-and-drop file upload zone (also clickable) accepting `.csv` files
- A progress indicator during upload and processing
- An import results summary showing inserted/updated/error counts after completion
- A table of import history fetched from `GET /api/import/history`

**Acceptance Criteria:**
1. Dragging a CSV file onto the drop zone uploads it to `POST /api/import/csv`.
2. A progress bar displays during upload.
3. After successful import, a summary card shows inserted/updated/error/skipped counts.
4. The import history table loads previous imports with timestamp, filename, and counts.
5. Uploading an invalid file shows an error message without crashing.

### L2-6.4: Search Page

The system shall provide a Search page with:
- A text input for natural-language queries
- Filter chips/dropdowns for Type, State, Priority, Module, Owner (populated from facets API)
- A results list showing requirement cards with similarity score, number, name, description snippet, and metadata badges
- A "Find Similar" button on each result card that triggers the similar-requirements API

**Acceptance Criteria:**
1. Entering a query and pressing Enter/Search calls `POST /api/search` and displays results.
2. Filter dropdowns are populated from `GET /api/requirements/facets`.
3. Each result card shows the similarity score as a percentage.
4. Clicking "Find Similar" on a result replaces the results with similar requirements for that item.
5. A "Clear" button resets the query and filters.

### L2-6.5: Requirements List Page

The system shall provide a Requirements page with:
- A paginated Material table of all requirements showing Number, Name, Type, State, Priority, Module
- Column sorting (client-side for current page, server-side for full sort)
- A detail panel/dialog that opens when clicking a row, showing all fields plus traced-to links and similar requirements
- A delete button with confirmation dialog

**Acceptance Criteria:**
1. The table loads requirements from `GET /api/requirements` with pagination.
2. Clicking a column header sorts the table by that column.
3. Clicking a row opens a detail view showing all requirement fields.
4. The detail view shows traced-to links as clickable chips that navigate to the linked requirement.
5. The delete button shows a confirmation dialog and calls `DELETE /api/requirements/{id}` on confirm.

## L1-7: Security & Air-Gap Compliance

### L2-7.1: Localhost-Only Binding

The ASP.NET Core application shall bind to `localhost` (127.0.0.1) only by default. The Qdrant instance shall be configured for localhost-only access. Kestrel configuration shall explicitly set `Urls` to `http://localhost:5000` in `appsettings.json`.

**Acceptance Criteria:**
1. The API is accessible at `http://localhost:5000` but not at `http://<machine-ip>:5000` from another machine.
2. The `appsettings.json` contains `"Urls": "http://localhost:5000"`.
3. Qdrant configuration binds only to localhost.

### L2-7.2: Network Egress Verification

The system shall not make any outbound HTTP/HTTPS calls. No NuGet package used shall require runtime network access for its core functionality. The ONNX model and tokenizer vocabulary are bundled as local files. A startup check shall verify the model file exists locally.

**Acceptance Criteria:**
1. Running the application with network disabled (no internet) still allows full functionality (import, embed, search, report).
2. No `HttpClient` instances are configured to call external URLs.
3. Application startup succeeds without network connectivity.

### L2-7.3: Audit Logging

The system shall log the following events to a local log file (Serilog, file sink):
- CSV import: timestamp, filename, user (if authenticated), record counts
- Search queries: timestamp, query text, result count
- Requirement deletion: timestamp, requirement number

Logs shall be written to a configurable local directory (default: `./logs/`). No log data shall be sent to external services.

**Acceptance Criteria:**
1. After an import, a log entry exists with timestamp, filename, and counts.
2. After a search, a log entry exists with the query text and result count.
3. Logs are written to the configured directory as daily rolling files.
4. No log sink is configured for any remote endpoint.

### L2-7.4: CORS Configuration

The system shall configure CORS to allow requests only from `http://localhost:4200` (Angular dev server) and the same-origin when served from the API host. No wildcard origins shall be permitted.

**Acceptance Criteria:**
1. The Angular dev server at `http://localhost:4200` can call the API without CORS errors.
2. A request from `http://evil.example.com` is rejected by CORS policy.
3. No `AllowAnyOrigin()` is present in the CORS configuration.
